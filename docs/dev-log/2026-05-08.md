## Gaps Identified

### Gap 1 — TaskScheduler Never Started (CONFIRMED ✅ FIXED)
The `TaskScheduler` was instantiated inside every `GAIANRuntime.__init__()` but
its `run_forever()` async loop was never launched at boot or on lazy runtime
creation. Tasks submitted to the queue sat dormant forever. `stats()` would
show `queued: N` permanently.

### Gap 2 — Dual Memory Write Divergence (CONFIRMED ✅ FIXED)
Every conversation turn wrote to two separate, non-communicating memory systems:
- `core/memory/memory_chroma.py` — ChromaDB, legacy, sentence-transformers
- `core/memory/store.py` — SQLite + sqlite-vec, Phase 3, full C17 audit

Memories written by the runtime were never retrieved by the router, and
vice versa. Both systems grew independently forever.

### Gap 3 — ActionGate Not Wired (IDENTIFIED — NOT YET FIXED)
`core/infra/action_gate.py` contains a clean GREEN/YELLOW/RED risk-tier veto
system with a human `confirm_callback` hook. It has never been instantiated,
imported into the request lifecycle, or connected to the Tauri frontend for
RED-tier confirmation dialogs. The `PolicyEngine` in the runtime is a separate
soft gate; the `ActionGate` is the hard infrastructure-level firewall.

### Task 1 — Scheduler Boot in `server_lifecycle.py`
**Commit:** [`7e9ec0c`](https://github.com/R0GV3TheAlchemist/GAIA-OS/commit/7e9ec0cdaf3c647d910975a10fe243a0fc5cbba3)

Added Step 3 to `_startup()`: calls `_boot_scheduler_for_existing_runtimes()` after
Viriditas completes. Added `_boot_scheduler_for_runtime(slug, rt)` — a guard-gated
launcher that checks `scheduler._running_flag` before calling
`asyncio.create_task(scheduler.run_forever())`. Upgraded `_shutdown()` to call
`rt._scheduler.stop()` for all runtimes and cancel/await background asyncio tasks
cleanly. `_SCHEDULER_TASKS` list tracks all background tasks for lifecycle management.

**What this activates:** Every `Task` submitted to any `GAIANRuntime`'s scheduler
now actually executes. Priority queue drains continuously. Policy gating, retries,
TTL expiry, and `on_success`/`on_failure` callbacks all fire. Goal Registry can
now be notified when tasks complete, closing the planner loop.

### Task 1b — Lazy-Init Scheduler Wiring in `_get_runtime()`
**Commit:** [`106b1ec`](https://github.com/R0GV3TheAlchemist/GAIA-OS/commit/106b1eccd3413a6d0ad7a08b0f33be8d6fe251db)

Added deferred import + call of `_boot_scheduler_for_runtime(slug, rt)` at the end
of the `if slug not in _RUNTIME_REGISTRY:` branch in `core/infra/server_state.py`.
Deferred import pattern used to avoid circular dependency:
`server_state → server_lifecycle → server_state`. Try/except wraps the call— non-fatal; runtime still works via `run_once()` if boot fails.

**What this closes:** Any GAIAN loaded lazily after server startup now gets its
scheduler loop booted immediately on first access, not just at startup time.

### Task 2 — Memory Unification Bridge
**Commit:** [`94a727d`](https://github.com/R0GV3TheAlchemist/GAIA-OS/commit/94a727d6b9fa042eae9549bce49f901c884fdb9b)

**New file:** `core/infra/memory_bridge.py`

Drop-in replacement for `core/memory/memory_chroma.py` convenience helpers.
Exposes identical API:
- `recall_for_prompt(query, gaian_slug, top_k=5) -> list[str]`
- `store_turn(user_message, gaian_response, gaian_slug, session_id, emotion) -> None`

Routes through `GAIANRuntime._memory_store` (SQLite + sqlite-vec, C17-governed)
instead of the ChromaDB singleton. ChromaDB retained as graceful fallback for
anonymous queries and cold-start scenarios where no runtime is registered.

Importance scoring: GAIAN responses stored at `0.60`, user messages at `0.55` —
small signal that synthesised model output has slightly higher recall value.

`kinds` filter on recall: `[MESSAGE, REFLECTION, FACT]` — excludes `GOAL` and
`TASK` items from prompt injection (planner internals, not conversational context).

**Updated file:** `core/routers/chat.py`

Single import line changed:
```python
# Before
from core.memory_chroma import recall_for_prompt, store_turn
# After
from core.infra.memory_bridge import recall_for_prompt, store_turn
```
All call sites, logic, and `done_data` fields identical. Zero behaviour regression.
