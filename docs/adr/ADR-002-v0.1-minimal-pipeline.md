# ADR-002 — v0.1 Minimal Viable Pipeline

**Status:** Accepted
**Date:** 2026-06-30
**Author:** GAIA Core

---

## Problem

GAIA's full architecture spans seven layers. Attempting to run and validate all seven
simultaneously creates a system that is impossible to test, debug, or demonstrate.
We need a runnable, verifiable minimal version that proves the core loop works.

## Decision

The v0.1 target is a single linear pipeline:

```
Input (string)
  → LLM Parser (extract structured claim)
  → Ontology (classify entity and relation types)
  → Epistemic Evaluation (assign confidence + knowledge_type)
  → World State (store claim)
  → Save Snapshot (persist)
  → Return: structured, persisted, typed knowledge
```

If this pipeline works end-to-end, GAIA is alive.
Every other layer (distribution, trust, simulation) is an upgrade on top of this core.

This is the definition of "done" for v0.1:
- A string input produces a persisted, epistemically-typed claim
- The snapshot can be reloaded and queried
- The pipeline is covered by automated tests
- The pipeline runs in under 2 seconds on a single machine

## Alternatives Considered

| Alternative | Why rejected |
|---|---|
| Build all 7 layers before any end-to-end test | No runnable system for months; architectural errors compound silently |
| Skip LLM parser, use structured input only | Reduces GAIA's core value proposition; adds it back later is harder than starting with it |
| Build distributed system first | Distribution is an upgrade on correct single-node behaviour, not a prerequisite |

## Consequences

**Gains:**
- Runnable system within days, not months
- Every subsequent layer can be validated against a working baseline
- Demonstrates GAIA's core loop to contributors immediately
- Architectural errors surface early when cost of fixing is low

**Losses / Trade-offs:**
- v0.1 has no distribution, no trust scoring, no simulation — not suitable for production
- LLM parser introduces an external dependency from day one

**Follow-on work:**
- `examples/minimal_pipeline.py` — reference implementation of this pipeline
- Issue: Add LLM parser integration tests with mocked responses
- ADR-003: LLM provider abstraction (swap providers without changing pipeline)
